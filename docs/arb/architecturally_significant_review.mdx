Use this template to document ARB architectural requirements.

| **In Scope (Y/N)** | **Requirement**                  | **Requirement Guidance**                                                             | **Project Planning Notes**                                                                                                                                                                                                  |
| ------------------ | -------------------------------- | ------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| N                  | SSO                              | Define application audiences and SSO/IDP strategy per audience.                      | Toolkit is invoked via GitHub Actions; no end-user consumer interface requiring SSO. GitHub Actions authentication via OIDC federation.                                                                                     |
| N                  | ADA                              | Consumer-facing applications should meet applicable WCAG accessibility requirements. | Toolkit produces no consumer-facing UI. Documentation portal (destination) maintains separate accessibility compliance.                                                                                                     |
| N                  | 2FA                              | Required for confidential/higher data or admin functions where applicable.           | GitHub Actions OIDC federation used; secrets managed via GitHub repository secrets.                                                                                                                                         |
| N                  | CCPA/CPRA                        | Consumer privacy controls and disclosures should be implemented when required.       | Toolkit transfers metadata and documentation; no consumer PII processed by toolkit itself. Downstream portal maintains privacy controls.                                                                                    |
| N                  | Browser/Responsive               | Define browser and device support based on audience.                                 | Toolkit is server-side workflow automation; no browser requirements.                                                                                                                                                        |
| Y                  | Open Source                      | Ensure OSS usage follows organizational compliance policy.                           | Toolkit uses GitHub Actions, Structurizr, and other internal reusable actions. All dependencies tracked via repository scanning.                                                                                             |
| Y                  | Disaster Recovery                | Define DR approach and any uncovered components.                                     | Toolkit stores results in `audi-red-documentation` repository (Git-backed). Daily scheduled runs provide recovery points. GitHub Actions retry logic handles transient failures. No external state; re-runs are idempotent. |
| Y                  | 3rd Party service levels         | Document SLA and support model for external providers.                               | GitHub Actions: standard GitHub availability SLA
| N                  | Data Retention Archiving Process | Define archival and retention responsibilities.                                      | Toolkit outputs archived in `audi-red-documentation/data/archive/` as per retention policy. Historical snapshots preserved for trend analysis.                                                                            |
| N                  | Cloud Hosting Risk               | Capture cloud vendor lock-in risk and required approvals.                            | Toolkit runs on GitHub Actions (GitHub Inc./Microsoft). Alternative: self-hosted runners or GitHub Enterprise. Portable workflow design allows migration.                                                                   |
| N                  | MultiLingual                     | Define language support requirements by market.                                      | Toolkit operates on English documentation and metadata. No multi-language requirement.                                                                                                                                      |
